# The recurring real RMF gap is SURFACES, not actions — "RMF can't open Settings" is a wrong assumption

Verified Jul 2026: the Apple client's `NavigationTarget` already includes `sync`, `settings`, `settings.general`, `duckai.settings`, `import.passwords`, `appearance`, `pir.main`, `feedback` (`apple-browsers/.../RemoteMessaging/Model/RemoteMessageModel.swift:356-366`), and iOS handles all but `softwareUpdate` (`iOS/DuckDuckGo/MessageNavigator.swift:50-73`) — so "we used a bespoke promo because RMF can't deep-link to Settings > Sync & Backup" (claimed on the promo-queue Asana task, Jul 9) is false; Cristian's correction ("wrong assumption, or missing conversation") is confirmed in code. What RMF genuinely lacks on iOS is *places to render*: only the NTP card (banner templates) and the What's New modal (`cardsList`) exist (`iOS/DuckDuckGo/DefaultRemoteMessagingSurfacesProvider.swift:25-32`) — no list headers, settings rows, menu badges/dots, pills/toasts, tab-switcher. Craig Russell's framing is the accurate one: RMF could be the coordinator only if it knew every surface, "and that's some distance away from what we have today." Also true: no RMF action can write a client setting (e.g. cookie-popup preference), which is a different, real gap.

Why it mattered: gap lists drive the iteration-2/3 promo decisions, and Mark explicitly demanded verified requirements/gaps before any approach change — repeating folklore gaps (actions) while missing structural ones (surfaces, single-scheduled-message, no caps) would steer the whole program wrong. When someone says "RMF can't do X", check `RemoteMessageModel.swift` + the surfaces provider before believing it.
