# The schema's `surfaces` field doesn't tell you what renders — the template→surface matrix is client-hardcoded per platform (and silently drops)

Effective surface = the message's declared `surfaces` array ∩ a per-platform, per-template allowlist compiled into the app (`RemoteMessagingSurfacesProviding`): iOS maps banner templates → `new_tab_page` ONLY and `cards_list` → `modal` ONLY (`iOS/DuckDuckGo/DefaultRemoteMessagingSurfacesProvider.swift:25-32`); macOS maps banners → NTP+tab_bar (tab bar renderer additionally accepts only survey-action `big_single_action`) and `cards_list` → `dedicated_tab`, which nothing on macOS ever fetches — so cards_list is declared-valid but unrenderable there. Declared-but-unsupported surfaces are dropped with only a client log; a message left with no supported surface is silently discarded (`JsonToRemoteMessageModelMapper.swift:211-231`). Repo CI (ajv schema check) can't catch any of this, and the schemas drift from the client (iOS schema missing `pproSubscriptionTier`/`ntpAfterIdleState`; `promo_single_action` client-blocked on macOS; no android/windows schemas in the repo at all).

Why it mattered: when authoring or reviewing configs in this repo, "it validates against the schema" ≠ "it will show" — you must check the target platform's surfaces provider and renderer for the template/surface/action combo (and the app version that introduced it). This also means "add a surface to RMF" is a 5-part change (schema enum + OptionSet + surfaces provider + renderer + store bitmask), which is the right unit when estimating surface gaps in the promo-queue iteration-2/3 work.
