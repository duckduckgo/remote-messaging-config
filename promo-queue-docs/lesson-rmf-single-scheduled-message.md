# RMF keeps exactly ONE scheduled message at a time — across ALL surfaces; "queueing" is emergent, not designed

The Apple client store (`apple-browsers/SharedPackages/BrowserServicesKit/Sources/RemoteMessaging/RemoteMessagingStore.swift:92-125,443-506`) saves the single config-order winner as `scheduled` and marks every other scheduled message `done` (done-and-never-shown rows are deleted). So an NTP banner and a What's New modal can never be live simultaneously, and the first-matching message monopolizes all surfaces until dismissed or auto-expired (`dismissAfterDaysShown`); the next eligible message only appears after dismissal → config `invalidate` → next refresh. There is no priority field (config-file order is priority, `RemoteMessagingConfigMatcher.swift:51-78`), no impression caps, no cross-message cooldowns, no start/end dates.

Why it mattered: this is the core structural constraint for the "make RMF the promo queue" discussion (promo-queue iteration 3) — any design where RMF arbitrates multiple concurrent promos starts with reworking the store/processor to per-surface (or multi-) scheduling. It also explains authoring behavior in this config repo: campaign windows are enforced by editing/removing messages, and ordering messages in the JSON is a real scheduling decision, not cosmetics.
