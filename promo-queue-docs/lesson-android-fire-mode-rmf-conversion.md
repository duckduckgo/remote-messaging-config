# Android's Fire Mode promo → RMF conversion is the unit-of-work template for migrating a bespoke promo

Approved TD (Asana `1216213068497872`, Jul 2, 2026; from the Fire Mode "Promo cards" thread where a bespoke promo shipped outside RMF and got challenged): the conversion took (1) one new action `fireTabsPromo` (JSON type + mapper + command opening the tab switcher in Fire mode); (2) two boolean matching attributes — `fireModeAvailable` in `matchingRules` (capability gate) and `fireModeUsed` in `exclusionRules` (engagement signal) — one plugin implements both; (3) a self-retire observer that sets `fireModeUsed` on first Fire-mode entry and immediately dismisses the active message (attribute re-evaluation alone would wait up to 24h for the next config download); (4) a hybrid surface: the tab-switcher banner stays natively rendered but is keyed to "FIRE_TABS is the active RMF message", inheriting its matching/exclusion/kill; (5) campaign window enforced server-side by config edits; (6) illustration via remote image URL (remote placeholders now supported). Notably, product *simplified the requirement* to fit RMF (dropped the "user burned tabs" trigger), and Chris Thelwell accepted the up-to-24h trigger latency.

Process: new matching attributes must be proposed/documented in the Remote Messaging Asana project (`1207619243206445`, "Matching Attribute" section, list `1207619413684520`) so other platforms can adopt them — example proposal: iOS `ntpAfterIdleState` (3-state attribute + reciprocal `messageShown` exclusions; "bake all states in now, adding values later is release-gated").

Why it mattered: it calibrates effort ("small" = action + attributes + message, days not weeks), demonstrates the escalation ladder actually working (why not RMF → extend → simplify → bespoke last), and the native-surface-keyed-to-RMF-state trick is the likely pattern for iOS promos on surfaces RMF doesn't render (menu dots, list headers) in promo-queue iteration 2.
